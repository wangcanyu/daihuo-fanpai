## F. 开发历程(完整心路,写文章的素材库)

### 第〇幕 · 起点是一个疑问,不是一个设计(2026-06 下旬)

作者手工复刻爆款带货视频能做到六七成相似,于是把流程交给 agent 做成 skill——结果 **agent 做的反而不如手工**。大多数人到这里的结论是"AI 还不行",作者的反应是把它当成一个 bug 去 debug:**为什么会不如手工?知识明明都在。**整个项目由这一个疑问逼出来,没有先画架构图。

### 第一幕 · 诊断:凶手不是模型(06-29)

翻开旧管线代码找到病根,两条:①分镜按**音频转写**切段,而不是按**画面硬切**——违背了自己积累的最大教训;②最致命的是,写提示词的函数全是**写死的位置模板+罐头描述**("女性手举产品表情夸张惊讶"),反推产物里那份精细的分镜分析**根本没被读**。手工时人脑读了那份分析,agent 用填充句兜底。
**核心洞察就此定型:病在"反推→写提示词"的转换环节丢细节、丢动作,不在生成模型。**后续所有设计都在守一条:把反推到的主体+动作+关键细节原样带进生成。

### 第二幕 · 一个案例的完整闭环(06-30,七子白香皂)

方法论决定:**不急着写 skill,先拿一个真实案例手动跑通全流程**。42.8s、30 个硬切的洗脸皂视频。
第一轮:硬切节奏复刻成功(旧管线做不到的核心能力打通)——但保真稀碎:反推漏了"橙色草本水"和"满脸白泥膜"(钩子的灵魂),每段人物长相漂移。**第二个关键教训:生成忠实执行了不准的脚本 = 垃圾进垃圾出,病根在解构不在即梦。**
修正闭环:前 3 秒密集法医解构 + 人物锚定图 + 双图锚定重生成 → 钩子逐帧复刻成功。四个机制(密集解构/锚定图/VLM评委/垃圾进垃圾出)全部实证有效。

### 第三幕 · 换心脏(06-30,Gemini → Seed 2.1 Pro)

用户实测豆包 Seed 2.1 Pro 反推比 Gemini 准,一次原生视频调用替代了"抽音频+转写+抽帧+帧分析+法医补救"五六步。实验发现关推理+流式后**快 17 倍且精度不掉**。反推引擎从此换心,封装成 `seed_reverse.py`——第一块固化的基石。

### 第四幕 · 迁移与"卖真"的顿悟(06-30,参阿婆→高小参)

把竞品海参视频迁移成自己品牌。关键顿悟:**卖"真"的产品(生鲜/食品),最优解不是让 AI 生成质感,而是让 AI "动"你的真实素材**——image2video 轻运镜,参刺质感以假乱真、包装文字全程清晰。product_role 路由(口播/hero真图/包装)由此诞生。83s 全片跑通。

### 第五幕 · 固化成 skill(07-01)

五块可插拔引擎 + 三个 JSON 契约 + SKILL/DESIGN/HANDOFF 三件套。设计哲学:**每块只通过 JSON 文件交接,换任何一块不动别处;每个"验证过的配置"都写下为什么,防后人(包括未来的 agent)乱改。**同日 B 模式:把 4 套千川付费课蒸馏成方法论弹药包,插在规划后只改台词字段。评委 judge.py 上线后发现有趣现象:忠实复刻分低是正常的——**原片结构本来就烂,方法论指导的重构 > 机械复刻**(后来跨类目 69/90 最高分实证)。

### 第六幕 · 实战抓虫(07-01~02)

用户亲自实战暴露真 bug:提示词说"拉开礼袋"却只传了裸海参图 → 即梦自由发挥编了个假包装。修成"按形态别名扫出该段所有产品形态、真图全传+锚图完备性报警"。跨类目验证(胶原蛋白口服液→海参)确立最后一块认知:**跨类目只有说服结构能复制,画面必须理解后重构,不存在机械映射表。**发布 GitHub。

### 第七幕 · 强模型复审(07-07)

用户点名让更强的模型(Fable)复审 Opus 建的 skill。13 脚本全读,揪出的最典型问题:**judge 的 `--target` 收了参数但从没把原片传给模型——保真度分是编的**。此外代理全退役、私人 endpoint 收编成公共模型名、host_desc 治跨段穿着漂移。教训:**agent 写的代码也需要 agent 复审,而且要换个更强的来审。**

### 第八幕 · 交付层革命(07-12)

用户提出真实痛点:成品是"拼死"的片子,进剪映还得重新切开调整——管线里明明有分段信息,却被拼接扔掉了再让人肉恢复。设想两个产物:接近成品的一步到位版 / 自动导入剪映草稿。
关键技术验证:剪映 10.7 保存草稿会加密,但**读明文草稿正常**——探针草稿(pyJianYingDraft)实测识别/打开/编辑/回存全通。`deliver.py` 落地:剪映草稿五轨预铺(视频逐段/配音/逐句字幕/贴字参考/空BGM),素材自包含。架构红线:**FULL.mp4 永远保持无字幕无 BGM 的中性状态**(它是评委输入和回归基准),观感加工全在下游。字幕精度顺手升级:TTS 逐句合成时本来就知道每句真实时长,记下来就是精确字幕轴(timing.json)。

### 第九幕 · 榴莲群戏,能力边界的一次冲锋(07-12)

当天下午用户甩来一条 109.6s、50 硬切的**三角色办公室情景剧**(老板质问员工出差报销 7 万 2 → 掏出泰国榴莲样品 → 试吃真香 → 砍价上直播)。此前所有案例都是单主播,这是管线第一次跑群戏。一天之内:
- **反推误判被人工纠出**:引擎给 7 镜打了 package_text,用户看原片说"没有任何包装文字"——抽帧实锤,全部降级。**人审闸口的价值:用户的领域直觉 > 引擎的标签。**
- **群戏补丁**:按 shotlist 判断每段出镜角色,逐段挂 1-3 张人物锚定图+逐角色人设;产品未登场的段剥离产品图**防剧透**(悬念在 45 秒)。
- **S12 试生成暴露幻觉**:画面多出一个凭空捏的员工 → 提示词加人数硬约束"自始至终只有这 3 个角色",重生成修复。
- **让 Seed 当质检员**:把带音轨的成片喂给 Seed 2.1 Pro 逐句核对"谁在张嘴"——多人轮流说话的口型分配全对(实际约九成,残余在剪映草稿里一键修)。
- **Ark 腿撞政策墙**:含人脸参考图(AI 人像也算)被火山隐私拦截,官方唯一解=换虚拟人像。**两条生成腿的真实分工就此探明:人脸段=即梦专属,纯产品段=双腿。**
- 积分中途断粮、充值续跑、judge 双视频超限 SSL 断连(当场修进引擎)。
- 最终:三看 44/90(剧情流按千川 rubric 打分天然低),保真度 63/100,产品还原 16/20 最高。**最大差距=真人喜剧表演的生动性——这是当前技术的天花板,不是管线的锅。**用户验收:"整体不错,不必追求过度完美。"

### 第十幕 · 收尾与传承(07-13)

README 中英重写、封面换成四格真实产出、加作者与联系方式;HANDOFF 更新;skill 迁移给 Hermes(rsync 同步脚本,铁律=改版只改 engine)。途中还翻了一次车:重启网关时 kill 错进程杀掉了整个 tmux 服务器,微信网关全灭——用自启脚本恢复,教训固化进记忆。**连事故都是资产。**

### 心路主线(文章可用的骨架)

1. **起点是疑问不是蓝图**:"为什么 agent 不如手工"——把失望当 bug 来 debug。
2. **凶手从来不是模型**:三次归因全指向管线(转换丢细节/解构不准/单主播模板不适配群戏),三次修完效果立现。
3. **先手动跑通一个案例,再固化**:每块引擎都是先在真实案例里验证过才写成脚本,没有一行"想当然"的代码。
4. **把脑子里的判断显式化**:完备性关卡、人审闸口、评委、doctor——手工时代的高频判断全部变成了可执行的关卡。
5. **强模型开路→固化→普通模型跑量**:每次强模型的临场发挥(法医解构/群戏补丁/政策定性)事后都固化进代码或坑索引,对模型智力的依赖持续下降。这是"skill"区别于"prompt"的本质。
6. **人的位置**:领域直觉(包装没字/悬念不能剧透/喜剧节奏)始终是机器给不了的输入;人审闸口不是流程负担,是最便宜的兜底。
7. **诚实面对天花板**:AI 演不出真人喜剧的微表情——承认它,把精修留给剪映草稿,而不是无限重抽。"不必追求过度完美"本身就是一种工程判断。
